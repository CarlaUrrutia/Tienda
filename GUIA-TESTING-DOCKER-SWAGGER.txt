GUIA COMPLETA - PROYECTO TIENDA

Este documento tiene 3 partes:
  Como levantar y probar TODO con Docker (recomendado)
  Como levantar y probar CADA microservicio por separado, sin Docker (desde el IDE o la terminal), uno por uno.
  URLs de Swagger (documentacion interactiva de cada API)

Credenciales de base de datos (MySQL):
  TODOS los microservicios usan lo mismo ahora (incluyendo "rol" y "cupon",
  que antes eran la excepcion):
    usuario: root
    contraseña: (vacia en local sin Docker) / root (con Docker, ver docker-compose.yml)

IMPORTANTE - Login (Spring Security):
  Todos los microservicios usan el usuario/contraseña que Spring Security genera
  solo cada vez que arrancas, y aparece en la consola asi:
    "Using generated security password: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
  Ese usuario es "user" y la contraseña CAMBIA cada vez que reinicias el microservicio.
  ("cupon" ya no tiene login fijo, tambien quedo con este comportamiento estandar).


PARTE 1: LEVANTAR TODO CON DOCKER

1) Abre una terminal en la carpeta raiz del proyecto (donde esta el archivo docker-compose.yml).

2) Construye las imagenes (la primera vez tarda varios minutos):
   docker compose build

3) Levanta todos los contenedores:
   docker compose up -d

4) Verifica que todo esta corriendo (columna STATUS debe decir "Up"):
   docker compose ps

5) Si algo no dice "Up", revisa su log especifico, ejemplo:
   docker compose logs -f cliente

6) Para apagar todo:
   docker compose down

7) Para apagar y borrar tambien los datos de MySQL (empezar de cero):
   docker compose down -v

Nota: dentro de Docker, en cada arranque el "generated security password" tambien va a cambiar, para los 16 microservicios por igual. Revisalo con "docker compose logs -f <servicio>" si necesitas hacer login.


PARTE 2: LEVANTAR CADA MICROSERVICIO POR SEPARADO (sin Docker)

Requisito: tener MySQL corriendo en tu equipo (puede ser tu MySQL local, o solo el contenedor de MySQL con: docker compose up -d mysql)

Para levantar un microservicio SIN Docker tienes 2 opciones:

  OPCION A con VS Code
    - Abre la carpeta del microservicio (ej: cliente/)
    - Busca la clase principal (ej: ClienteApplication.java)
    - Click derecho -> Run / Ejecutar

  OPCION B (desde la terminal, con Maven):
    - Entra a la carpeta del microservicio, ejemplo:
        cd cliente
    - Ejecuta:
        mvn spring-boot:run
    - Cuando en la terminal veas algo como:
        "Started ClienteApplication in X seconds"
      significa que el microservicio LEVANTO correctamente. Esto es el testing: si arranca sin stacktrace de error rojo, el microservicio esta OK.
    - Si en cambio ves un error tipo "Communications link failure" o "Connection refused", el microservicio SI levanto bien, pero no encontro el MySQL.

A continuacion, el detalle de los 16 microservicios: como levantarlo, su URL base REST, su URL de Swagger (la real: /doc/swagger-ui/index.html), y ejemplos reales para CREAR, VER POR ID y ELIMINAR POR ID.

1) CLIENTE
Levantalo en: ClienteApplication (carpeta cliente/)
URL base:     http://localhost:8082/api/clientes
Swagger:      http://localhost:8082/doc/swagger-ui/index.html

Crear (POST):
curl -X POST http://localhost:8082/api/clientes ^
  -H "Content-Type: application/json" ^
  -d "{\"nombre\":\"Juan\",\"apellido\":\"Perez\",\"email\":\"juan@test.com\"}"

Ver por id (GET), reemplaza 1 por el id que te devolvio el POST:
curl http://localhost:8082/api/clientes/1

Eliminar por id (DELETE):
curl -X DELETE http://localhost:8082/api/clientes/1

NOTAS FINALES

- El simbolo ^ al final de linea en los curl es para Windows (CMD). Si usas Mac/Linux o Git Bash, cambia el ^ por \ al final de cada linea, o simplemente escribe el comando en una sola linea.

- Si prefieres no usar la terminal, puedes copiar cada bloque "Crear / Ver por id / Eliminar por id" directo en Postman, o directo en el propio Swagger UI (boton "Try it out" en cada endpoint).

- El orden 1 a 15 de esta lista NO es arbitrario: respeta las dependencias reales (ej. no puedes crear una Venta sin que exista antes un Cliente y un Empleado). Si pruebas fuera de orden y te da error 500 o "not found", revisa que el id que estas usando (id_cliente, id_producto, etc.) realmente exista.

- "rol" (antes "gerente") y "cupon" ya quedaron con las mismas credenciales root que el resto, tanto en local como en Docker. El docker-compose.yml ya sobreescribe usuario/contraseña de TODOS los servicios a root/root via variables de entorno, asi que no hace falta nada especial para estos dos.

- El docker-compose.yml se actualizo con las nuevas dependencias entre microservicios:
  - "empleado" ahora tambien depende de "rol" (variable MS_ROL_URL=http://rol:8080)
  - "tienda" ahora tambien depende de "empleado" (variable MS_EMPLEADO_URL=http://empleado:8081)
  Como "empleado" ya dependia de "tienda", agregar la dependencia inversa crearia
  un ciclo que Docker Compose no permite en el orden de arranque (depends_on).
  Por eso "tienda" SI tiene la variable de entorno para llamar a "empleado", pero
  NO espera a que "empleado" arranque primero (no es necesario: Feign solo la usa
  cuando alguien pega a un endpoint que la necesita, no al iniciar).

- OJO: el servicio "rol" en el docker-compose.yml asume que tu carpeta se llama
  "rol" (build: ./rol). Si tu carpeta todavia se llama "gerente", solo cambia esa
  linea a "build: ./gerente" antes de correr "docker compose build".
